title: Web App Tester
description: Automated web app behaviour verification triggered from a GitHub PR or Issue using Playwright CLI (headless Chromium). Finds the testable URL, runs a structured test plan, and posts a step-by-step report as a GitHub comment.
---

The **Web App Tester** verifies web app behaviour for a GitHub PR or Issue by running browser-based tests via **Playwright CLI** (headless Chromium).

It is **PR/Issue-driven**: point it at a PR or issue number and it finds the testable preview URL from comments, uses the existing test plan (or generates one automatically), executes each step in a fresh headless browser session, and posts a structured test execution report directly on the PR or issue.

Single command: **`/test-web-app`**

---

## How It Works

```mermaid
flowchart TD
    A[/test-web-app pr N or issue N] --> B[Fetch PR/Issue content, comments, commits, linked issues]
    B --> C{Test URL found in comments?}
    C -- No --> D[Post 'no URL found' comment and STOP]
    C -- Yes --> E{Test plan found?}
    E -- Yes --> F[Use existing test plan]
    E -- No --> G[Auto-generate test plan and post comment]
    F --> H{Chromium already cached?}
    G --> H
    H -- Yes --> I[Skip install]
    H -- No --> J[Install Chromium via npx]
    I --> K[Generate Playwright test script]
    J --> K
    K --> L[Execute script — headless Chromium, all steps in one session]
    L --> M[Parse JSON results]
    M --> N[Clean up temp files]
    N --> O[Post test execution report as GitHub comment]
```

1. **Gather** — fetches full PR/issue body, all comments, commit messages, and linked issue content via `gh` CLI.
2. **Find URL** — scans content for a testable URL with labels like `Preview URL:`, `Staging URL:`, `Deploy preview:`. Posts a comment and stops if none is found.
3. **Find or generate test plan** — uses an existing structured test plan from comments (including ones generated by `test-strategist`), or auto-generates one from the PR/issue context and posts it before execution.
4. **Prepare browser** — checks if Chromium is already cached; installs it only if missing (first run only, ~30–60s).
5. **Execute** — generates a single headless Playwright script covering all test steps, runs it once, retries failed steps up to 3 times with 2-second waits.
6. **Report** — parses JSON results, cleans up temp files, and posts a single structured comment on the PR/issue.

## Step Statuses

| Status | Meaning |
|---|---|
| ✅ PASSED | Step executed, expected outcome observed |
| ❌ FAILED | Step executed, expected outcome NOT observed |
| 🔴 BLOCKED | Step could not execute after 3 retries, or skipped due to production URL |

Overall result is **PASSED** when all steps pass, **FAILED** when steps fail but all were attempted, and **BLOCKED** when one or more steps could not execute.

---

## Auto-Generated Test Plan

If no test plan is found in the PR/issue, the agent generates one covering:
- The primary user-facing change
- Implied UI interactions (forms, navigation, state changes, error states)
- At least one happy-path and one edge/negative scenario

The auto-generated plan is posted as a GitHub comment before execution begins.

---

## Safety Rules

- **No URL found** → posts a comment and stops immediately
- **Production URL** (no `staging`, `preview`, `dev`, `test`, `localhost`, or PR/issue number in domain) → read-only mode; form submissions and destructive actions are skipped and marked BLOCKED
- **Report is strictly bounded** — the posted comment contains only the defined results table and failure details; no suggested fixes, recommendations, or commentary are ever added
- **Credentials** — never included in any posted comment
- **Temp files** — always deleted after the run (`_wat_test.cjs`, `_wat_results.json`, `_wat_screenshot_*.png`), even if execution fails

## Report Format

The plugin posts a single GitHub comment with this exact structure — nothing more:

```
🤖 web-app-tester — Test Execution Report
URL tested: https://preview.example.com
Total: 5 | ✅ Passed: 4 | ❌ Failed: 1 | 🔴 Blocked: 0
Overall: FAILED

| # | Step | Status |
|---|------|--------|
| 1 | Navigate to /login | ✅ PASSED |
| 2 | Fill email and password, click Sign In | ✅ PASSED |
| 3 | Verify redirect to /dashboard | ✅ PASSED |
| 4 | Click New Order button | ❌ FAILED |
| 5 | Verify order form appears | ✅ PASSED |

---

### Failed / Blocked Steps

**Step 4 — Click New Order button**
Reason: Button not found after 3 retries — element not in DOM
Screenshot: captured at point of failure
```

## Performance

| Phase | First run | Subsequent runs | Xianix executor (pre-installed) |
|---|---|---|---|
| Browser install | ~30–60s | 0s (cached) | 0s always |
| Script generation | ~5s | ~5s | ~5s |
| Test execution (9 steps) | ~20s | ~20s | ~20s |
| **Total** | **~60–90s** | **~25s** | **~25s** |

Browser caching is automatic — the plugin checks for a cached Chromium before attempting any install.

For the **Xianix platform**: request that `playwright` and Chromium be pre-installed in `99xio/xianix-executor:latest` to eliminate the first-run install cost entirely.

## Prerequisites

- **Node.js 20+** — runs the generated Playwright test script
- **`gh` CLI** installed and authenticated (`gh auth login` or `GITHUB_TOKEN` env var)

See [docs/setup.md](docs/setup.md) for full setup instructions.

## Environment Variables

| Variable | Required | Purpose |
|---|---|---|
| `GITHUB_TOKEN` | Yes (if `gh auth login` not run) | Authenticate `gh` CLI for reading PRs/issues and posting comments |

## Quick Start

```bash
# Navigate to the GitHub repo you want to test
cd /path/to/your-repo

# Launch Claude Code with the plugin
claude --plugin-dir /path/to/xianix-plugins-official/plugins/web-app-tester

# Trigger a test run
/test-web-app pr 42
```

Make sure your PR has a comment with a testable URL, for example:
```
Preview URL: https://pr-42--my-app.netlify.app
```

## Xianix Platform Integration

The plugin is designed for the Xianix executor environment (Docker container, Python 3.12 + Node.js 20). It uses only:
- `gh` CLI (pre-installed in the executor)
- `node` / `npx` (pre-installed via Node.js 20)
- `Bash` — no MCP server required

No additional environment configuration is needed beyond `GITHUB_TOKEN`.
